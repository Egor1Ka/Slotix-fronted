# Staff Schedule Management — Design Spec

## Обзор

Две страницы для управления расписанием сотрудников в контексте организации:

1. **Страница сотрудника** — редактирование своего расписания, выходных, просмотр букингов
2. **Админская страница** — просмотр расписания любого сотрудника (readonly) с фильтром

## Подход

Подход A: две независимые страницы с общими компонентами. Общие таб-компоненты принимают `staffId`, `orgId`, `readOnly` и переиспользуются на обеих страницах.

---

## Роуты

| Роут                                   | Доступ                               | Назначение                                                       |
| -------------------------------------- | ------------------------------------ | ---------------------------------------------------------------- |
| `/(org)/org/[orgId]/my-schedule`       | member, admin, owner (как сотрудник) | Своё расписание — 3 таба с редактированием                       |
| `/(org)/manage/[orgId]/staff-schedule` | admin, owner                         | Просмотр расписания любого сотрудника — фильтр + 3 readonly таба |

### Навигация

- Для сотрудника: ссылка в сайдбаре организации
- Для админа: новый пункт "Staff Schedule" в `OrgSidebar` рядом с Positions и Services

### Доступ

- Backend проверяет роль: member видит только своё, admin/owner — любого сотрудника
- member не имеет доступа к `/manage/[orgId]/staff-schedule`
- admin/owner могут зайти на `/org/[orgId]/my-schedule` чтобы видеть своё расписание как сотрудник

## Общие компоненты

Расположение: `components/staff-schedule/`

Все три таб-компонента принимают общие пропсы:

```ts
interface StaffScheduleTabProps {
	staffId: string
	orgId: string
	readOnly: boolean
}
```

### 1. ScheduleViewTab

Отображает недельное расписание (пн-вс) с временными слотами.

- **readOnly=false** (сотрудник): переиспользует существующий `ScheduleEditor` — выбор дней недели, установка часов работы, сохранение через `scheduleApi.updateTemplate`
- **readOnly=true** (админ): те же данные в режиме только просмотр, без кнопок редактирования

**Данные:** `scheduleApi.getTemplate(staffId)`

### 2. OverridesTab

Список всех schedule overrides сотрудника, разделённый на две секции:

- **Будущие** — предстоящие выходные/переопределения
- **Прошедшие** — история выходных

Каждый override отображает: дату, причину (если указана), временные слоты (если частичный выходной).

- **readOnly=false** (сотрудник): кнопка "Добавить выходной" → форма `ScheduleOverrideForm`. Можно удалить будущий выходной. Прошедшие — только просмотр.
- **readOnly=true** (админ): только просмотр всего списка

**Данные:** новый эндпоинт `GET /api/schedule/overrides?staffId=xxx` (список всех overrides) + `DELETE /api/schedule/override/{id}` (удаление)

### 3. BookingsTab

Плоский список букингов, сгруппированный по датам (заголовки-разделители как в мессенджерах).

Каждый букинг отображает:

- Время (начало — конец)
- Имя клиента
- Название услуги
- Статус (бейдж с цветом)

Клик на букинг → Sheet справа с полными деталями. Переиспользуем части из `BookingDetailPanel`/`BookingDetailsPanel`.

Всегда readonly на обеих страницах — букинги не редактируются отсюда.

**Диапазон дат:** по умолчанию загружает текущую неделю. Кнопки навигации "← Предыдущая неделя / Следующая неделя →" для подгрузки других периодов.

**Данные:** `bookingApi.getByStaff(staffId, dateFrom, dateTo)`

## Админская страница

**Роут:** `/(org)/manage/[orgId]/staff-schedule`

### Layout

```
┌─────────────────────────────────┐
│ [🔍 Выбрать сотрудника ▾]      │
├─────────────────────────────────┤
│ [Расписание] [Выходные] [Букинги]│
├─────────────────────────────────┤
│                                 │
│   Контент выбранного таба       │
│   (readOnly)                    │
│                                 │
└─────────────────────────────────┘
```

### Фильтр сотрудника

- Select/Combobox с поиском для выбора сотрудника
- Показывает аватар + имя + должность
- Данные: `orgApi.getStaff(orgId)`
- staffId сохраняется в URL query param (`?staffId=xxx`) для возможности шаринга ссылки
- Активный таб тоже в URL query param (`?staffId=xxx&tab=overrides`)
- Если staffId не выбран — пустое состояние "Выберите сотрудника"

### Табы

Все 3 таба в режиме `readOnly=true`:

1. **Расписание** — просмотр недельного расписания сотрудника
2. **Выходные** — просмотр всех overrides (будущие + прошедшие)
3. **Букинги** — просмотр букингов, кликабельные для деталей

## Страница сотрудника

**Роут:** `/(org)/org/[orgId]/my-schedule`

### Layout

```
┌─────────────────────────────────┐
│ Моё расписание — "Org Name"     │
├─────────────────────────────────┤
│ [Расписание] [Выходные] [Букинги]│
├─────────────────────────────────┤
│                                 │
│   Контент выбранного таба       │
│   (editable для расписания      │
│    и выходных)                  │
│                                 │
└─────────────────────────────────┘
```

### Заголовок

- "Моё расписание" + название организации
- Нет фильтра — сотрудник видит только себя
- staffId определяется из сессии (текущий пользователь → его staff member ID в этой организации)

### Табы

1. **Расписание** — `readOnly=false` — полное редактирование через `ScheduleEditor`
2. **Выходные** — `readOnly=false` — создание/удаление будущих overrides, просмотр прошедших
3. **Букинги** — `readOnly=true` — просмотр своих букингов

## API

### Существующие эндпоинты (хватает)

| Эндпоинт                                                      | Метод | Назначение                     |
| ------------------------------------------------------------- | ----- | ------------------------------ |
| `/api/schedule/template?staffId={id}`                         | GET   | Получить расписание сотрудника |
| `/api/schedule/template`                                      | PUT   | Обновить расписание            |
| `/api/schedule/override`                                      | POST  | Создать выходной/override      |
| `/api/bookings/by-staff?staffId={id}&dateFrom={d}&dateTo={d}` | GET   | Букинги сотрудника             |
| `/api/org/{id}/staff`                                         | GET   | Список сотрудников организации |

### Нужные эндпоинты (проверить на бэке)

| Эндпоинт                               | Метод  | Назначение                       |
| -------------------------------------- | ------ | -------------------------------- |
| `/api/schedule/overrides?staffId={id}` | GET    | Список всех overrides сотрудника |
| `/api/schedule/override/{id}`          | DELETE | Удалить override                 |

### Маппинг user → staffId

Для страницы сотрудника нужен способ определить staffId текущего пользователя в конкретной организации. Варианты (определится при имплементации в зависимости от того что поддерживает бэкенд):

- Через `orgApi.getUserOrgs()` — если возвращает staffId
- Через `orgApi.getStaff(orgId)` — найти по userId текущего пользователя
- Отдельный эндпоинт `GET /api/org/{orgId}/me` — возвращает staff member текущего пользователя

### API конфиги (фронтенд)

Создать `services/configs/schedule-override.config.ts` с эндпоинтами для overrides. Остальные конфиги уже существуют в `lib/booking-api-client.ts`.

```

├── ScheduleViewTab.tsx        — Таб расписания (ScheduleEditor wrapper)
├── OverridesTab.tsx           — Таб выходных (список + форма)

├── org/[orgId]/my-schedule/
```

## Переиспользование существующих компонентов

| Компонент                                    | Где переиспользуется                               |
| -------------------------------------------- | -------------------------------------------------- |
| `ScheduleEditor`                             | Внутри `ScheduleViewTab` (readOnly=false)          |
| `ScheduleOverrideForm`                       | Внутри `OverridesTab` (readOnly=false)             |
| `BookingDetailPanel` / `BookingDetailsPanel` | Части переиспользуются в Sheet при клике на букинг |
| `Tabs` (shadcn)                              | Обёртка для 3 табов на обеих страницах             |
| `Sheet` (shadcn)                             | Для деталей букинга                                |
| `Select` / `Combobox` (shadcn)               | Для фильтра сотрудника на админской странице       |
| `Badge` (shadcn)                             | Для статуса букинга                                |
| `Empty` (shadcn)                             | Для пустых состояний                               |

---

## Что НЕ входит в скоуп

- Редактирование букингов со страниц расписания (только просмотр)
- Создание выходных админом за сотрудника (только просмотр)
- Уведомления о изменениях расписания
- Мобильная адаптация (базовая responsive через Tailwind, без отдельного mobile дизайна)
